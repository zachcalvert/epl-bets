# Phase 10: Production Hardening & Observability

## Status: Planned

**Depends on:** Phase 9 (Deploy & Publish) — complete
**Ref:** See `LAUNCH-PLAN.md` §Phase 10

## Goal

Make the live deployment easier to operate and safer to trust by improving visibility, resilience, and production safeguards.

## Why This Phase Now

The project has crossed the line from “works locally” to “publicly deployed.” The next highest-leverage work is making production behavior easier to inspect, debug, and maintain without losing the portfolio-friendly simplicity of the stack.

## Proposed Scope

### 10A — Operational Visibility

- Add structured logging across Django requests, Celery tasks, and websocket events
- Capture production exceptions from web and background workers in a shared error-reporting tool
- Add explicit health checks for the web app and any worker-critical dependencies
- Document where to look first when deploys, live scores, or odds syncs fail
- Add a small production runbook for non-interactive Fly maintenance commands and common recovery steps

### 10B — Runtime Guardrails

- Audit production settings for cookies, CSRF, allowed hosts, proxy/SSL handling, and secret management
- Add abuse protection on auth-sensitive and state-changing endpoints
- Clarify migration/seed expectations during deploy so startup is deterministic
- Define cleanup or retention rules for stale operational data
- Document baseline Fly resource sizing by process group so SSH sessions and management commands do not OOM the live machine

### 10C — Performance & Resilience

- Profile the dashboard, odds board, match detail, and my bets pages for query count and slow template/data paths
- Introduce targeted caching where freshness tradeoffs are acceptable
- Improve degraded-mode behavior when external APIs time out or return incomplete data
- Create a lightweight post-deploy smoke checklist
- Validate memory headroom for the `app`, `worker`, and `beat` groups after deploy and after seed/migration workflows

## Deliverables

- production runbook doc
- Fly sizing recommendations for each process group
- health-check endpoint(s) and monitoring notes
- logging/error-reporting configuration
- caching/performance pass on the highest-traffic pages
- deploy verification checklist

## Open Decisions

1. Which monitoring/error-reporting tool gives enough value without overcomplicating a portfolio app?
2. Should operational telemetry live entirely in logs/admin, or should some of it also power product-facing transparency features later?
3. Should seed data ever run automatically in production after initial bootstrap?
4. Which caches are acceptable given that fixtures, odds, and standings have different freshness expectations?
5. Should the project keep a single Fly app with multiple process groups, or split background work into a separate deployment if memory pressure keeps growing?

## Suggested Implementation Order

1. Add health checks and logging first so later changes are easier to verify.
2. Tighten production settings and deploy behavior.
3. Profile the key pages and add the smallest useful caching layer.
4. Write the smoke-check/runbook docs last, once the operational path is settled.
