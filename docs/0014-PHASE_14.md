# Phase 14: Bankruptcy & Bailout System

## Status: Planning

**Depends on:** Phase 5 (Betting Flow) вҖ” complete

## Goal

When a player goes broke, keep the game fun. Instead of a dead end, give them a dramatic, tongue-in-cheek bailout experience that resets their balance and gets them back in the action.

## Why This Phase

The app uses fake credits with no real-money stakes. A player running out of credits shouldn't feel like a punishment вҖ” it should feel like a comedic beat. This phase turns bankruptcy into a memorable gameplay moment:

- demonstrates full-stack polish: models, context processors, atomic transactions, CSS animations, vanilla JS
- exercises the same balance-management patterns as betting and rewards, reinforcing architectural consistency
- keeps every user permanently engaged вҖ” no one is ever "stuck"
- gives portfolio reviewers a fun UX detail that shows attention to player experience

## Core Concepts

### Bankruptcy

A user is **bankrupt** when:

1. Their balance is below the minimum bet (0.50 cr), **AND**
2. They have **zero pending bets**

The second condition is critical вҖ” a user who still has action on the table isn't bankrupt yet. Their pending bets may settle in their favor.

### Bailout

A bailout is a one-time credit injection that resolves a bankruptcy. When a user requests a bailout:

- A 10-second countdown plays with humorous flavor text ("The EPL Bets Central Bank is reviewing your application...")
- A random amount between **1,000вҖ“3,000 cr** is credited to their balance
- The amount varies each time, adding a small element of surprise

### The Overlay

When a bankrupt user loads any page, a **full-screen overlay** blocks the entire site. They cannot navigate, place bets, or interact with anything until they request a bailout. This creates a clear "game over вҶ’ continue?" moment.

## Models

### `Bankruptcy`

| Field | Type | Notes |
|-------|------|-------|
| `user` | ForeignKey(User) | The bankrupt user |
| `balance_at_bankruptcy` | DecimalField | Snapshot of remaining balance when bankruptcy was declared |
| `created_at` | DateTimeField(auto_now_add) | When the bankruptcy occurred |

### `Bailout`

| Field | Type | Notes |
|-------|------|-------|
| `user` | ForeignKey(User) | The bailed-out user |
| `bankruptcy` | OneToOneField(Bankruptcy) | Each bankruptcy gets exactly one bailout |
| `amount` | DecimalField | Random amount between 1,000вҖ“3,000 that was credited |
| `created_at` | DateTimeField(auto_now_add) | When the bailout was granted |

Both models live in the `betting` app alongside `UserBalance` and `BetSlip`.

## Technical Approach

### Context processor (`betting/context_processors.py`)

New file. For authenticated users, checks:

```python
is_bankrupt = balance < 0.50 and pending_bet_count == 0
```

Adds to every template context:
- `is_bankrupt` вҖ” boolean, drives overlay rendering
- `bankruptcy_count` вҖ” integer, total past bankruptcies for display

Registered in `settings.py` TEMPLATES context_processors.

### Bailout view

`POST /betting/bailout/` вҖ” `BailoutView`

Server-side validation:
- User is authenticated
- Balance < 0.50
- No pending bets

In an atomic transaction with `select_for_update()` on UserBalance:
1. Create `Bankruptcy` record (balance snapshot)
2. Generate random amount: `random.randint(1000, 3000)`
3. Create `Bailout` record linked to the bankruptcy
4. Credit `UserBalance.balance += amount`

Returns JSON: `{ "success": true, "amount": 2150, "new_balance": 2150.00 }`

The client shows the approved amount, then reloads the page after a brief delay.

### Overlay template

`betting/templates/betting/partials/bailout_overlay.html`

Included in `base.html`:
```django
{% if is_bankrupt %}
    {% include "betting/partials/bailout_overlay.html" %}
{% endif %}
```

Fixed position, z-index above everything. Three states managed by vanilla JS:
1. **Request state** вҖ” heading, balance, bankruptcy count, bailout button
2. **Countdown state** вҖ” bank review message, ticking number, cycling flavor text
3. **Result state** вҖ” approved message, credited amount, auto-reload

### JavaScript

Inline in the overlay partial (small, self-contained). Handles:
- Button click вҶ’ start countdown
- `setInterval` for 1-second ticks
- Flavor text rotation every ~2 seconds
- `fetch()` POST to bailout endpoint at countdown end
- `window.location.reload()` after showing result

### CSS animations

Added to `styles.css`:
- `.bailout-overlay` вҖ” fixed positioning, backdrop blur, dark semi-transparent bg
- `.bailout-countdown` вҖ” scale/pulse keyframe on each tick
- Fade transitions between the three overlay states

## Data Flow

```
Bet settlement (or bet placement) вҶ’ balance drops below 0.50
  вҶ’ All pending bets settled (none remaining)
    вҶ’ Next page load: context processor detects bankruptcy
      вҶ’ base.html renders overlay (blocks entire site)
        вҶ’ User clicks "Request Bailout"
          вҶ’ 10-second client-side countdown with flavor text
            вҶ’ POST /betting/bailout/
              вҶ’ Server: create Bankruptcy + Bailout, credit balance
                вҶ’ Client: show approved amount, reload page
                  вҶ’ Context processor: balance is healthy, no overlay
```

### Admin: simulate bankruptcy

A custom admin action on the User changelist: **"Simulate bankruptcy"**. For each selected user:

1. Settles all their pending bets as losses (status=LOST, payout=0)
2. Sets their balance to 0.00

This lets the site operator (or developer during testing) trigger the full bankruptcy вҶ’ bailout experience on demand without touching match data. The action is purely a testing/demo convenience вҖ” it doesn't create Bankruptcy records itself (those are created naturally when the user requests a bailout).

## Testing Expectations

- **Model tests:** Bankruptcy/Bailout creation, foreign key relationships
- **Context processor tests:** bankrupt when balance < 0.50 and no pending bets; NOT bankrupt when pending bets exist; NOT bankrupt when balance >= 0.50
- **View tests:** bailout endpoint credits correct amount, creates both records atomically, rejects request when user has pending bets or sufficient balance
- **Edge cases:** concurrent bailout requests (select_for_update), user with no UserBalance record

## Non-Goals

- No social/public shaming вҖ” bankruptcy count is visible only to the user
- No cooldown between bailouts вҖ” if they go broke again immediately, they can bail out again
- No bailout limits вҖ” this is a fun demo, not a real economy
- No admin controls for bailout amounts вҖ” hardcoded range is fine for V1

## Definition of Done

Phase 14 is complete when:

- Users with balance < 0.50 and no pending bets see a full-page overlay on every page
- The overlay cannot be dismissed without requesting a bailout
- The bailout countdown runs for 10 seconds with animated flavor text
- A random 1,000вҖ“3,000 cr is credited and the page reloads with the overlay gone
- Bankruptcy and Bailout records are persisted with correct relationships
- The bankruptcy count is displayed on the overlay
- Tests cover the detection logic, bailout transaction, and edge cases
