# Phase 13: Rewards System

## Status: Planning

**Depends on:** Phase 12 (Transparency & Portfolio Depth) — complete

## Goal

Give the site operator (via Django admin) the ability to distribute credit rewards to all users or a targeted subset. When a user receives a reward, the UI should surface it with a fun, gratifying experience that makes the fake-betting loop feel more alive.

## Why This Phase

The app already tracks user balances and has a settlement flow that adjusts credits. A rewards system adds a new admin-driven mechanic that:

- demonstrates Django admin customization (custom actions, filters, bulk operations)
- creates a new real-time UI moment (reward notification) that exercises the existing WebSocket/HTMX stack
- makes the demo feel more like a living product — balances change not just from bets but from operator actions
- gives portfolio reviewers another touchpoint showing the full request lifecycle: admin action -> Celery task -> balance update -> WebSocket push -> animated UI

## Core Concepts

### Reward (admin-defined)

A reward is a named credit grant with an amount. Examples: "Welcome Bonus", "Matchday 12 Giveaway", "Top Bettor Bonus".

Each reward has:
- `name` — human-readable label (e.g. "Weekend Bonus")
- `amount` — credit value to add to each recipient's balance
- `description` — optional copy shown to the user in the notification
- `created_by` — FK to admin user who created it
- `created_at` — timestamp

### RewardDistribution (per-user record)

Each user who receives a reward gets a distribution record:
- `reward` — FK to Reward
- `user` — FK to User
- `created_at` — inherited from BaseModel
- `seen` — whether the user has dismissed/acknowledged the reward in the UI

This model serves double duty: it's the ledger of who got what, and the `seen` flag drives the UI notification state.

## Admin Experience

### Creating & distributing rewards

Two approaches to explore during implementation:

**Option A — Two-step: create then distribute**
1. Admin creates a Reward in the standard model admin
2. From the Reward change page, a custom admin action "Distribute to all users" or "Distribute to selected users" triggers distribution
3. For targeted distribution, admin could select users from User changelist and run a "Grant reward" action

**Option B — Single-step: inline distribution**
1. Admin creates a Reward and picks recipients (all / specific users) in the same form
2. On save, distribution happens automatically

Either way, the distribution step should:
- Create RewardDistribution records for each recipient
- Credit each recipient's UserBalance.balance
- Optionally trigger a WebSocket push so online users see the reward immediately

### Admin filters and display

- Reward list: name, amount, recipient count, created_by, created_at
- RewardDistribution list: filterable by reward, user, seen status
- User changelist: custom action to grant an existing reward to selected users

## User Experience

### Reward notification

When a user receives a reward, the UI should feel celebratory:

- A toast/modal/banner appears with the reward name, amount, and optional description
- Visual flair: confetti animation, glow effect, or similar lightweight celebration
- The notification should feel distinct from the existing toast system — this is a special moment
- The user's balance updates in real time (navbar balance display)
- A dismiss/acknowledge action marks the distribution as `seen`

### Delivery mechanisms

- **Online users (WebSocket):** push the reward notification immediately via the existing channel layer
- **Offline users (next page load):** on any page load, check for unseen RewardDistributions and show the notification
- The notification should only show once — dismissing it sets `seen = True`

### Rewards history (optional stretch)

A small section on the My Bets page or a dedicated tab showing past rewards received. Low priority for initial implementation.

## Technical Approach

### New models (in `betting` app or new `rewards` app — TBD)

- `Reward` — name, amount, description, created_by, created_at
- `RewardDistribution` — reward FK, user FK, credited_at, seen

### Admin customization

- Custom admin actions on User changelist and Reward changelist
- Potentially a custom admin view for the distribution flow if the UX warrants it

### Balance crediting

- Reuse existing `UserBalance` update patterns from bet settlement
- Wrap in a transaction: create RewardDistribution + increment balance atomically

### Real-time delivery

- After crediting, broadcast to each recipient's WebSocket channel (if connected)
- The consumer sends an OOB HTMX swap with the reward notification HTML
- Fallback: middleware or context processor checks for unseen rewards on page load

### Celebration UI

- Keep it CSS/HTML-first, consistent with the app's no-JS-framework philosophy
- Confetti: lightweight CSS keyframe animation or a tiny vanilla JS snippet
- The reward notification component should be a reusable template include
- Auto-dismiss after a timeout, or manual dismiss via HTMX delete/patch

## Resolved Decisions

1. **New `rewards` app.** Keeps concerns clean; will import `UserBalance` from `betting` where needed.
2. **Inline distribution (no Celery task).** Demo app with few users — admin action credits balances synchronously.
3. **Enhanced toast** for the celebration UI. Distinct from the standard toast but not a blocking modal.
4. **No reward expiry.** Keep it simple for V1.

## Slices

### 13A — Models & Admin

- Reward and RewardDistribution models
- Migrations
- Admin registration with custom actions for distribution
- Balance crediting logic with transaction safety

### 13B — Notification UI

- Reward notification template component (celebration animation)
- Unseen reward check on page load (context processor or middleware)
- HTMX-driven dismiss action
- Balance refresh in navbar after reward

### 13C — Real-Time Push

- WebSocket delivery for online users when reward is distributed
- OOB swap to inject the notification without page reload
- Integration with existing consumer infrastructure

## Testing Expectations

- Model tests: reward creation, distribution record creation, balance crediting
- Admin tests: custom action triggers distribution correctly
- View/template tests: unseen reward renders notification, dismiss marks seen
- WebSocket tests: reward broadcast reaches connected user
- Edge cases: user with no balance record, duplicate distribution prevention, concurrent crediting

## Non-Goals

- No user-to-user gifting
- No reward marketplace or redemption
- No complex eligibility rules (bet X times to unlock) — that's a loyalty program, not a rewards drop
- No email/push notification — in-app only

## Definition of Done

Phase 13 is complete when:

- Admin can create a reward and distribute it to all or selected users from Django admin
- Each recipient's balance is credited atomically
- Online users see a celebratory notification in real time via WebSocket
- Offline users see the notification on their next page load
- Users can dismiss the notification, and it does not reappear
- Tests cover the distribution flow, balance updates, and notification lifecycle
