# Phase 11: Homepage Leaderboard

## Status: Planned

**Depends on:** Phase 10 (Production Hardening & Observability) — planned
**Ref:** See `LAUNCH-PLAN.md` §Phase 11

## Goal

Add a live leaderboard to the homepage that highlights the ten users with the highest balances and refreshes from the backend every 30 seconds.

## Why This Phase Now

The app already has the core fake-betting loop: signup, balances, bet placement, and settlement. A leaderboard turns that existing data into a more social and demo-friendly surface without adding a new product domain or pulling the stack away from server-rendered Django + HTMX patterns.

It also gives the landing experience a clearer sense of activity. Reviewers can immediately see that users exist, balances change over time, and the backend can drive periodic UI refreshes without a JavaScript framework.

## Proposed Scope

### 11A - Leaderboard Data Query

- Rank users by `UserBalance.balance` descending
- Limit the result set to the top 10 users
- Include stable tie-breaking so the order does not flicker between refreshes
- Return only lightweight fields needed for display, such as email and current balance
- Decide whether to mask email addresses slightly for presentation/privacy polish

### 11B - Homepage UI

- Add a leaderboard card/section to the root dashboard page at `/`
- Keep the existing match dashboard content intact and place the leaderboard where it feels complementary rather than crowded
- Show rank, user identity label, and formatted balance
- Include a strong empty state for fresh environments where seeded users or balances do not exist yet

### 11C - HTMX Polling Refresh

- Render the leaderboard through a reusable partial
- Add an HTMX endpoint that returns just the leaderboard body
- Poll the backend every 30 seconds with `hx-get` and `hx-trigger="every 30s"`
- Ensure the first page render and the polling response share the same template path so the markup stays consistent

### 11D - Test Coverage

- Add view tests for initial homepage render with leaderboard data
- Add tests for the partial endpoint and its ordering/limit behavior
- Add a test for the empty state when no balances exist
- Add a tie-case test so ordering is deterministic

## Deliverables

- homepage leaderboard section on the dashboard
- leaderboard partial + polling endpoint
- queryset logic for top balances
- tests covering render, refresh, limit, and empty-state behavior
- updated planning docs reflecting the new Phase 11 direction

## Suggested Implementation Shape

### Views

- Extend `matches.views.DashboardView` or extract shared leaderboard context logic into a small helper/service
- Add a dedicated partial view, likely in `matches` because `/` already lives there

### Templates

- Update `matches/templates/matches/dashboard.html`
- Add a partial such as `matches/templates/matches/partials/leaderboard.html`

### Routing

- Add a route under `matches/urls.py` for the polling partial, for example `/leaderboard/`

## Open Decisions

1. Should the leaderboard identify users by full email, masked email, or a derived display name?
2. Should rank ties be broken by `updated_at`, `created_at`, or user id for maximum stability?
3. Should the leaderboard be visible to anonymous visitors, authenticated users, or everyone?
4. Should we also show the current signed-in user's rank if they are outside the top 10, or keep Phase 11 strictly to the top 10 list?

## Suggested Implementation Order

1. Define the ranking query and deterministic tie-breaker.
2. Add the reusable leaderboard partial and render it on the dashboard.
3. Add the HTMX polling endpoint and wire the section to refresh every 30 seconds.
4. Add focused tests for ordering, limit, partial rendering, and empty-state behavior.
