# Phase 11: Homepage Leaderboard

## Status: Complete

**Depends on:** Phase 10 (Production Hardening & Observability) — planned
**Ref:** See `LAUNCH-PLAN.md` §Phase 11

## Goal

Add a live leaderboard to the homepage that highlights the ten users with the highest balances and refreshes from the backend every 30 seconds.

## Why This Phase Now

The app already has the core fake-betting loop: signup, balances, bet placement, and settlement. A leaderboard turns that existing data into a more social and demo-friendly surface without adding a new product domain or pulling the stack away from server-rendered Django + HTMX patterns.

It also gives the landing experience a clearer sense of activity. Reviewers can immediately see that users exist, balances change over time, and the backend can drive periodic UI refreshes without a JavaScript framework.

## Shipped Scope

### 11A - Leaderboard Data Query

- Ranked users by `UserBalance.balance` descending
- Limited the result set to the top 10 users
- Added deterministic tie-breaking by user id so the order does not flicker between refreshes
- Reused lightweight display fields suitable for homepage rendering
- Masked public email addresses on the leaderboard for better presentation/privacy

### 11B - Homepage UI

- Added a leaderboard card to the root dashboard page at `/`
- Kept the existing live match dashboard intact and placed the leaderboard alongside it
- Showed rank, masked user identity label, and formatted balance
- Included an empty state for fresh environments without user balances
- Added a signed-in "your rank" callout on the homepage when the user is outside the top 10

### 11C - HTMX Polling Refresh

- Rendered the leaderboard through a reusable partial
- Added an HTMX endpoint that returns just the leaderboard body
- Polled the backend every 30 seconds with `hx-get` and `hx-trigger="every 30s"`
- Shared the same template path between the first page render and the polling response

### 11D - Test Coverage

- Added view tests for initial homepage render with leaderboard data
- Added tests for the partial endpoint and its ordering/limit behavior
- Added empty-state coverage when no balances exist
- Added tie-case coverage so ordering is deterministic
- Added coverage for masked emails and the signed-in user rank behavior

## Deliverables

- homepage leaderboard section on the dashboard
- leaderboard partial plus polling endpoint
- shared leaderboard/rank helper logic
- masked public email display
- "your rank" homepage callout that links into `My Bets`
- `My Bets` rank summary section for signed-in users
- tests covering render, refresh, limit, tie-breaking, masking, and rank behavior

## Outcome

Phase 11 shipped as a compact but visible product enhancement. The homepage now feels more alive, the backend-driven HTMX refresh pattern is demonstrated on the landing experience, and signed-in users have a clearer path from the public leaderboard into their own account performance in `My Bets`.
