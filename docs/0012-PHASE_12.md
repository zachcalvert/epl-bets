# Phase 12: Transparency & Portfolio Depth

## Status: Complete

**Depends on:** Phase 11 (Homepage Leaderboard) — complete  
**Ref:** Original transparency direction, shipped as the first Under the Hood rollout

## Goal

Deepen the portfolio value of EPL Bets by making real system activity visible inside the product itself. The phase should help a reviewer understand what Django, HTMX, Redis, Channels, Celery, and PostgreSQL are doing without reading the whole codebase first.

## Product Positioning

Phase 12 is not a generic analytics phase and not a production-ops dashboard. It is a guided transparency layer for a demo app:

- show real backend-driven activity already happening in the system
- explain it in plain language at the moment it is relevant
- keep the implementation mostly server-rendered and aligned with existing HTMX patterns
- improve the portfolio story for engineering reviewers first, while staying understandable to broader viewers

## Resolved Decisions

### 1. Panel placement

The "Under the Hood" surface should be page-specific, available on flagship pages, and collapsed by default.

- Start on `Dashboard`, `Match Detail`, and `Odds Board`
- Use a consistent toggle and layout so it feels global without forcing noise onto every screen
- Do not hide it behind a separate demo mode for the initial release
- Link from the panel into `/how-it-works/` for deeper explanation

### 2. Audience priority

Optimize for engineering reviewers first.

- The copy should explain real mechanisms, not only product behavior
- Avoid jargon walls: every event should have a short human summary plus optional technical detail
- Portfolio viewers who are less technical should still be able to follow the "what happened" layer

### 3. Visualization approach

Stay HTML/CSS first, with small progressive enhancement acceptable only when the server-rendered version remains complete.

- Event feeds, status cards, and summary metrics should render fully from Django templates
- Lightweight SVG is acceptable for simple trend visuals
- Avoid introducing a charting library or client-heavy state model in this phase

### 4. Event scope

Only surface events that are already meaningful to the product story:

- HTMX partial refreshes
- WebSocket score broadcasts
- Celery task activity relevant to currently viewed data
- bet placement and bet settlement outcomes
- targeted cache hit/miss or data freshness indicators where the explanation is trustworthy

Do not surface raw internal noise such as every ORM query, every middleware hop, or full log streams.

## Shipped Scope

### 12A - Event Timeline Foundation

Shipped a shared event model that the UI can read regardless of whether the source is a request, background task, or websocket broadcast.

#### Event model

Each event should carry a compact, display-ready payload:

- `id`
- `occurred_at`
- `category` such as `htmx`, `websocket`, `celery`, `betting`, `cache`
- `source` such as `dashboard`, `match_detail`, `odds_board`, `fetch_live_scores`
- `action` such as `partial_refreshed`, `score_broadcast`, `odds_synced`, `bet_settled`
- `summary` for human-readable UI copy
- `detail` for optional technical context
- `route` or `entity_ref` where relevant, such as match id
- `status` such as `info`, `success`, `warning`

#### Storage strategy

Implemented Redis/cache-backed short-lived event storage.

- Keep a small rolling list per page scope and a small global list for recent system activity
- Retain only a limited recent window suitable for a demo surface
- Avoid a permanent database model unless the first iteration proves a need for historical reporting

#### Initial event producers

- middleware or view helpers for HTMX partial responses
- Celery tasks for `fetch_live_scores`, `fetch_odds`, and `settle_match_bets`
- websocket broadcast points in `matches.tasks` when score updates are pushed
- betting flow hooks around successful bet placement and settlement

### 12B - Under the Hood UI

Added a collapsible Under the Hood panel on the flagship pages.

#### Shipped behavior

- Show a short status summary at the top, such as "Odds refreshed 12s ago" or "Last live score push 8s ago"
- Render the last 5 to 10 relevant events in reverse chronological order
- Group visually by category so HTMX, WebSocket, and Celery activity are easy to scan
- Include "why this matters" copy for the most important event types

#### Shipped page coverage

- `Dashboard`: websocket live-score broadcasts, leaderboard HTMX polling, current data freshness
- `Match Detail`: websocket score updates, odds polling refreshes, bet placement result, settlement when applicable
- `Odds Board`: periodic odds refreshes, API freshness, stale-data fallback state

### 12C - Outcome

The most important portfolio goal is now live in-product:

- `Dashboard`, `Match Detail`, and `Odds Board` each expose a scoped Under the Hood panel
- panels are fed by real recorded events rather than static explainer text
- HTMX refreshes, websocket score broadcasts, Celery syncs, and betting lifecycle events are visible in context
- the UI stays server-rendered and HTMX-driven, consistent with the rest of the app

## Follow-On Work

The broader transparency theme still has room to grow, but it is no longer blocking Phase 12 completion.

### Future Architecture Storytelling

Expand `/how-it-works/` so it connects the abstract architecture diagram to the real event feed users can see elsewhere.

- Add a deployment/runtime section covering Fly, web/worker/beat roles, Redis, and PostgreSQL
- Add concrete walkthroughs for three paths: page render, odds polling, live score broadcast
- Cross-link each walkthrough back to the relevant live page
- Add a short glossary for HTMX polling, channel layers, and background jobs

### Future Demo-Friendly Insights

Add one or two compact insights that make the fake-betting loop feel more alive without turning this into a BI feature set.

Recommended first candidates:

- settlement snapshot on `My Bets`: won/lost/void counts plus most recent settlement event
- odds freshness / line movement callout on `Odds Board` or `Match Detail`

Defer richer historical ROI charts unless the simpler surfaces land well first.

## Deliverables

- reusable Under the Hood panel component
- shared event recording helper plus Redis-backed storage
- event producers wired into HTMX, websocket, and Celery touchpoints
- scoped Under the Hood panels on the three flagship pages
- category-aware event feed cards and summary panels
- tests covering event recording, event trimming, task emission, and panel rendering

## Implementation Notes

### Slice 1 - Event plumbing

1. Add a small transparency service module for recording and reading recent events.
2. Define scope keys for global events and per-page/per-match events.
3. Instrument the smallest high-signal producers:
   - leaderboard and odds HTMX partials
   - `fetch_live_scores`
   - websocket score broadcast
   - `fetch_odds`
   - successful bet placement
4. Add unit tests around event payload generation and event list trimming.

### Slice 2 - First panel release

1. Ship the panel on `Dashboard` only.
2. Show recent events plus one top summary card.
3. Poll the panel via HTMX so it updates without custom frontend state.
4. Validate that the panel stays useful even when no live match is active.

### Slice 3 - Match and odds expansion

1. Extend panel support to `Match Detail` and `Odds Board`.
2. Add page-specific summaries for odds freshness and live-score activity.
3. Add betting-related events where they improve the story, not where they add repetition.

These three shipped slices established the core of the feature. Architecture-storytelling expansion and extra insight surfaces remain optional future work rather than unfinished requirements of this phase.

## Testing Expectations

- service tests for event recording, trimming, and scope retrieval
- view tests for panel render, empty states, and HTMX partial refreshes
- task tests asserting high-value events are emitted on score, odds, and settlement flows
- template assertions for explanation copy on flagship pages

## Non-Goals

- no full observability dashboard for operators
- no persistent historical analytics warehouse
- no frontend framework or charting library adoption
- no attempt to expose every internal system event

## Risks To Manage

1. Noise: too many low-value events will make the panel feel gimmicky.
2. Coupling: event capture must stay lightweight and not tangle core business logic.
3. Accuracy: freshness and cache messaging must only be shown when backed by real data.
4. Performance: panel polling should reuse cheap queries and bounded event lists.

## Definition Of Done

Phase 12 is complete when:

- at least three flagship pages expose a useful Under the Hood panel
- the panel is driven by real recorded events rather than hard-coded explanatory text
- targeted tests cover the new event plumbing and the three panel surfaces

## Outcome

Phase 12 shipped successfully as the first Under the Hood rollout. The app now demonstrates its Django, HTMX, Redis, Channels, Celery, and betting flows from inside the product itself, which materially improves the portfolio story without introducing a heavier frontend architecture.
