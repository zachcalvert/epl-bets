# Phase 12: Transparency & Portfolio Depth

## Status: Backlog

**Depends on:** Phase 11 (Homepage Leaderboard) — in progress  
**Ref:** Preserved from the original Phase 11 draft

## Goal

Deepen the project's portfolio value by making the system behavior more legible, more interesting to explore, and more memorable for reviewers.

## Why This Phase Still Matters

The leaderboard adds visible activity to the homepage, but the original transparency direction is still valuable. It strengthens the project's portfolio story by making the Django, HTMX, Redis, Channels, and Celery architecture easier to understand from inside the product itself.

## Proposed Scope

### 12A - Under the Hood Panel

- Build a collapsible panel that explains what just happened on the page
- Show signals such as HTMX request/response, websocket updates, Celery task activity, and cache usage when available
- Keep the implementation server-rendered and HTMX-friendly instead of drifting into a SPA toolchain

### 12B - Guided Architecture Story

- Expand `/how-it-works/` with production deployment details and more concrete request/data-flow walkthroughs
- Add tighter cross-links from live pages into architecture explanations
- Make the portfolio narrative clearer for recruiters, hiring managers, and technical reviewers

### 12C - Demo-Oriented Product Enhancements

- Add richer fake-betting insights such as ROI trend summaries, settlement snapshots, or line-movement highlights
- Create a first-visit walkthrough path so evaluators quickly see live scores, odds polling, and bet settlement behavior
- Improve explanation copy around fake-money constraints, odds freshness, and where real external data enters the system

## Deliverables

- "Under the Hood" UI and supporting backend event/context plumbing
- expanded architecture/storytelling content
- one or more demo-friendly analytics or insight surfaces
- updated README/screenshots for portfolio presentation

## Open Decisions

1. Should the "Under the Hood" panel be global, page-specific, or limited to a dedicated demo mode?
2. Which internal events are worth surfacing without adding noise or leaking implementation details users do not care about?
3. Should charts and visualizations stay HTML/CSS/SVG only, or is small progressive enhancement acceptable here?
4. Which audience should this phase optimize for first: engineering reviewers or broader portfolio viewers?

## Suggested Implementation Order

1. Define the event model for "Under the Hood" so the UI is driven by real system activity.
2. Add the smallest compelling version of the panel on one or two flagship pages.
3. Expand the architecture and portfolio storytelling around that foundation.
4. Layer in demo insights once the narrative structure is working.
